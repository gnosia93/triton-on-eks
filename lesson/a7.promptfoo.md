## promptfoo 로 모델 평가하기 ##

LLM 애플리케이션을 개발하다 보면 이런 상황을 자주 마주하게 된다. 프롬프트를 수정했는데 기존보다 나아진 건지 확신이 없고, 모델을 바꿔보고 싶은데 하나하나 수동으로 비교하기엔 시간이 너무 걸린다. GPT-4o와 Claude를 놓고 어떤 게 우리 서비스에 더 맞는지 판단하려면, 같은 질문을 양쪽에 던지고 결과를 눈으로 비교하는 노가다를 반복해야 한다.
promptfoo는 이 과정을 자동화하는 오픈소스 LLM 평가 프레임워크다. 한마디로 표현하면, LLM을 위한 단위 테스트 프레임워크라고 할 수 있다.
소프트웨어 개발에서 코드를 변경할 때마다 단위 테스트를 돌려서 기존 기능이 깨지지 않았는지 확인하듯이, promptfoo는 프롬프트나 모델을 변경할 때마다 평가를 돌려서 품질이 유지되는지 확인할 수 있게 해준다.

### 동작 방식 ###
사용법은 단순하다. YAML 파일 하나에 세 가지를 정의한다.

* 프롬프트: 테스트할 프롬프트 템플릿
* 프로바이더: 비교할 모델 목록 (OpenAI, Anthropic, AWS Bedrock 등 90개 이상 지원)
* 테스트 케이스: 입력값과 평가 기준

### 평가 기준 ###
promptfoo가 강력한 이유는 다양한 평가 방식을 지원한다는 점이다.
결정적(Deterministic) 평가는 코드로 판정하는 방식이다. 특정 문자열 포함 여부(contains), JSON 형식 검증(is-json), 응답 시간 제한(latency), 비용 제한(cost) 등이 있다. 정확한 기준이 있을 때 유용하다.  
모델 기반(Model-assisted) 평가는 LLM이 다른 LLM의 출력을 채점하는 방식이다. llm-rubric이 대표적인데, 자연어로 채점 기준을 작성하면 채점용 LLM이 해당 기준에 따라 pass/fail을 판정한다. "답변이 친절한가?", "기술적으로 정확한가?" 같은 주관적 기준도 평가할 수 있어서, 사람이 일일이 채점하는 수고를 줄여준다.

### 어떤 상황에서 유용한가? ###
* 프롬프트 엔지니어링: 프롬프트를 수정할 때마다 기존 품질이 유지되는지 자동 검증
* 모델 비교: 동일한 테스트셋으로 여러 모델의 품질, 비용, 속도를 객관적으로 비교
* CI/CD 통합: 프롬프트 변경을 코드 변경처럼 관리하고, PR마다 자동 평가 실행
* 레드팀 테스트: 67개 이상의 보안 공격 플러그인으로 LLM 취약점 스캔

promptfoo 을 이용하여 AWS Bedrock 에서 제공되는 모델의 성능을 측정한다. 

### 1. 사전준비 ###
npm(Node Package Manager) 을 이용하여 promptfoo 및 AWS Bedrock SDK 를 설치한다.
```
npm install promptfoo
npm install @aws-sdk/client-bedrock-runtime

brew install node
node --version
```

### 2. AWS 인증 설정 ###
아래와 같이 AWS 인증 정보를 AWS CLI로 설정 또는 환경 변수로 설정한다. 
```
aws configure
```
또는
```
export AWS_ACCESS_KEY_ID="your_access_key"
export AWS_SECRET_ACCESS_KEY="your_secret_key"
export AWS_DEFAULT_REGION="us-east-1"
```

### 3. 지원 모델 확인 ###
AWS CLI 를 이용하여 설정된 리전을 확인한 후, 지원되는 모델 리스트를 확인한다. 
```
aws configure get region
aws bedrock list-foundation-models --query "modelSummaries[*].modelId" --output table
```

### 4. 프로젝트 초기화 ###
```
mkdir llm-eval
cd llm-eval
npx promptfoo init
```

모델 채점용(llm-rubric) 으로 haiku 모델을 사용할 예정이다. 아래 명령어로 동작여부를 체크한다.
```
 aws bedrock-runtime invoke-model \
  --model-id apac.anthropic.claude-3-haiku-20240307-v1:0 \
  --region ap-northeast-2 \
  --content-type application/json \
  --body '{"anthropic_version":"bedrock-2023-05-31","max_tokens":100,"messages":[{"role":"user","content":"hello"}]}' \
  /tmp/response.json
```
[결과]
```
{
    "contentType": "application/json"
}
```


### 5. 테스트 시나리오 작성 ###
```
cat <<EOF > promptfooconfig.yaml
description: AWS Bedrock 모델 비교 평가

prompts:
  - |
    다음 질문에 한국어로 답변해줘.
    정확하고 간결하게 답변할 것.

    질문: {{question}}

providers:
  # Claude Sonnet 4
  - id: bedrock:apac.anthropic.claude-sonnet-4-20250514-v1:0
    config:
      region: ap-northeast-2
      temperature: 0
      max_tokens: 1024

  - id: bedrock:apac.anthropic.claude-3-haiku-20240307-v1:0
    config:
      region: ap-northeast-2
      temperature: 0
      max_tokens: 1024

  # Amazon Nova Pro
  - id: bedrock:apac.amazon.nova-pro-v1:0
    config:
      region: ap-northeast-2
      interfaceConfig:
        temperature: 0
        max_new_tokens: 1024

# 채점용 LLM도 Bedrock으로 (OpenAI API 키 불필요)
defaultTest:
  options:
    provider:
      id: bedrock:apac.anthropic.claude-3-haiku-20240307-v1:0
      config:
        region: ap-northeast-2
        temperature: 0
  assert:
    # cost 는 지원하지 않으므로 주석처리한다. 
    #- type: cost
    #  threshold: 0.05
    - type: latency
      threshold: 10000
    # 한국어로 답변했는가
    - type: llm-rubric
      value: "답변이 한국어로 작성되어 있는가?"
      metric: 언어
    # 기술적 정확성
    - type: llm-rubric
      value: "기술적으로 틀린 내용이 없는가?"
      metric: 정확성
    # 구조화된 답변인가
    - type: llm-rubric
      value: "답변이 논리적으로 구조화되어 있고, 핵심을 먼저 말하는가?"
      metric: 구조
    # 실용성
    - type: llm-rubric
      value: "실무에서 바로 활용할 수 있는 구체적인 예시나 코드가 포함되어 있는가?"
      metric: 실용성
    # 간결성 (너무 길지 않은가)
    - type: javascript
      value: "output.length < 3000 ? 1.0 : Math.max(0, 1 - (output.length - 3000) / 5000)"
      metric: 간결성

tests:
  - vars:
      question: "쿠버네티스에서 파드와 디플로이먼트의 차이점은?"
    threshold: 0.7
    assert:
      - type: llm-rubric
        value: "파드와 디플로이먼트의 차이를 정확히 설명하고, 실무 관점의 예시가 포함되어 있는가?"
    
  - vars:
      question: "REST API와 GraphQL의 장단점을 비교해줘"
    threshold: 0.7
    assert:
      - type: llm-rubric
        value: "두 기술의 장단점이 균형 있게 설명되어 있는가?"
      - type: contains
        value: "GraphQL"

  - vars:
      question: "Docker 컨테이너와 가상머신의 차이는?"
    threshold: 0.7
    assert:
      - type: llm-rubric
        value: "기술적으로 정확하고, 초보자도 이해할 수 있는 설명인가?"
EOF
```

### 6. 테스트 실행 ###
```
npx promptfoo@latest eval

```
