## LangGraph로 Agentic AI 만들기 ##

RAG 파이프라인은 `질문 → 벡터 검색 → 리랭킹 → LLM 답변` 흐름을 동작한다. 질문 하나에 검색 한 번, 답변 한 번. 단일 턴, 정적 흐름이다. 이 구조는 "우리 회사 휴가 규정 알려줘" 같은 단순 질의에는 잘 동작하지만 실제 사용자의 질문은 그렇게 단순하지 않다.
"Llama 3.1 405B 서빙에 필요한 GPU 구성이 궁금한데, 최근 6개월 내 벤치마크 기준으로 알려줘. vLLM이랑 TensorRT-LLM도 비교해줘." 에 대한 질문에 대해서 RAG가 이 질문에 제대로 답하려면

* 질문을 쪼개야 한다: 메모리 계산, GPU 선정, 프레임워크 비교 → 세 개의 서브 질문
* 검색을 여러 번 해야 한다: 각 서브 질문마다 다른 키워드로
* 최신성을 판단해야 한다: "최근 6개월"이라는 조건을 실제로 적용해야 함
* 검색 결과가 부족하면 다시 찾아야 한다: 한 번 검색해서 안 나오면?
* 여러 소스를 종합해야 한다: 논문 + 블로그 + 벤치마크 결과

사실 단일 턴 RAG는 이 중 아무것도 수행하지 못한다. 주어진 질문을 한 번 검색하고, 나온 문서로 한 번 답변할 뿐이다.

### Agentic AI ###
Agentic은 LLM을 "한 번 답하는 존재" 에서 "스스로 판단하며 여러 단계를 밟는 존재" 로 바꾼다.
```
질문 → [LLM이 판단] → 어떤 도구를 쓸지 결정
                 ↓
              도구 실행 (검색, 계산, API 호출)
                 ↓
           [LLM이 관찰] → 결과가 충분한가?
                 ↓
        충분하면 답변, 부족하면 다시 도구 호출
```
핵심은 루프다. LLM이 스스로 "아직 부족하다"를 판단하고, 추가 행동을 결정하고, 이 루프가 있어서 복잡한 질문도 단계적으로 풀 수 있다.

Agentic의 가장 널리 쓰이는 설계 패턴은 ReAct로, 이름 그대로 추론(Reason)과 행동(Act)을 번갈아 수행한다.
```
Reason:  "이 질문에 답하려면 먼저 벤치마크 논문을 찾아야겠다"
   ↓
Act:     paper_search("vLLM vs TensorRT-LLM benchmark")
   ↓
Observe: 논문 3편을 받았다. 그중 한 편이 2024년 발표, 적합함
   ↓
Reason:  "블로그에서 실측 수치도 보강해야겠다"
   ↓
Act:     web_search("Llama 405B inference H100 benchmark 2024")
   ↓
Observe: 블로그 2개와 벤치마크 레포를 받았다
   ↓
Reason:  "이제 답변하기에 충분하다"
   ↓
최종 답변 생성
```
이 패턴의 가치는 투명성이다. LLM이 왜 그 도구를 선택했고, 결과를 어떻게 해석했는지 단계마다 드러난다. 디버깅이 가능해지고, 로그로 품질을 평가할 수 있다.

### LangGraph는 에이전트 개발을 위한 프레임워크로 ... ###

* 상태(State) 관리: 그래프 실행 중 공유되는 데이터 (메시지, 중간 결과 등)
* 체크포인트(Checkpointer): 상태를 저장해 중단/재개, 시간 여행 가능
* 조건부 흐름: LLM의 판단에 따라 다른 경로로 분기
* Human-in-the-loop: 특정 노드에서 사람 승인을 기다릴 수 있음
* 관측성: 모든 전이가 추적 가능, Langfuse 연동 자연스러움  

"Agentic 루프를 안전하게 운영하기 위한 인프라" 이다.

## 에이전트 개발 ##

### 시나리오 ###
```
사용자: "요즘 LLM 추론 최적화 트렌드가 뭐야?"
     ↓
에이전트:
  1. 질문 분석 → 검색 전략 수립
  2. [tool] paper_search(RAG MCP)로 관련 논문 찾기
  3. [tool] web_search로 최신 블로그/뉴스 보강
  4. 결과 비교/종합
  5. 필요하면 후속 검색 (인용 논문 추적)
  6. 요약 + 출처 정리해서 답변
```

### 2. 상태와 메모리 ###
  - LangGraph State: 그래프 전체에서 공유되는 상태
  - 단기 메모리: 대화 컨텍스트 (MessagesState)
  - 장기 메모리: 사용자 프로필, 선호도 (외부 저장소)
  - 실습: 이전 대화 기억하는 에이전트

### 4. 세션과 영속성 ###
  - Checkpointer: 대화 상태 저장/복구
  - 세션 ID 관리, 타임아웃
  - 실습: Redis/Postgres 체크포인터 붙이기
